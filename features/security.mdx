Security is a top priority in Conar. All sensitive data is encrypted, credentials are never stored in plaintext, and connections use industry-standard security protocols.

## Encryption

Conar uses military-grade encryption to protect your database credentials and sensitive information.

### AES-256-GCM Encryption

All connection strings and passwords are encrypted using **AES-256-GCM**:

- **Algorithm**: AES (Advanced Encryption Standard) with 256-bit keys
- **Mode**: GCM (Galois/Counter Mode) for authenticated encryption
- **Key length**: 256 bits (32 bytes)
- **IV length**: 12 bytes (96 bits)
- **Salt length**: 16 bytes (128 bits)

### Key Derivation

Encryption keys are derived using **PBKDF2** (Password-Based Key Derivation Function 2):

```typescript
const key = crypto.pbkdf2Sync(
  secret,      // Your master password
  salt,        // Unique random salt
  1,           // Iteration count
  KEY_LENGTH,  // 32 bytes
  'sha256'     // Hash algorithm
)
```

<Note>
Each encrypted value has its own unique salt, ensuring that identical values produce different ciphertext.
</Note>

### Encryption Implementation

Here's how Conar encrypts your connection strings:

```typescript
// Simplified encryption flow
function encrypt({ text, secret }) {
  // Generate random IV and salt
  const iv = crypto.randomBytes(12)
  const salt = crypto.randomBytes(16)
  
  // Derive encryption key from secret
  const key = crypto.pbkdf2Sync(secret, salt, 1, 32, 'sha256')
  
  // Create cipher and encrypt
  const cipher = crypto.createCipheriv('aes-256-gcm', key, iv)
  const encrypted = Buffer.concat([
    cipher.update(text, 'utf8'),
    cipher.final(),
    cipher.getAuthTag()  // Authentication tag for integrity
  ])
  
  // Return as concatenated hex string
  return `${iv.toString('hex')}.${encrypted.toString('hex')}.${salt.toString('hex')}`
}
```

### Decryption Process

```typescript
function decrypt({ encryptedText, secret }) {
  // Split encrypted string into components
  const [ivHex, encryptedHex, saltHex] = encryptedText.split('.')
  
  // Convert from hex to buffers
  const iv = Buffer.from(ivHex, 'hex')
  const encrypted = Buffer.from(encryptedHex, 'hex')
  const salt = Buffer.from(saltHex, 'hex')
  
  // Extract auth tag (last 16 bytes)
  const authTag = encrypted.slice(-16)
  const ciphertext = encrypted.slice(0, -16)
  
  // Derive same key using salt
  const key = crypto.pbkdf2Sync(secret, salt, 1, 32, 'sha256')
  
  // Create decipher and decrypt
  const decipher = crypto.createDecipheriv('aes-256-gcm', key, iv)
  decipher.setAuthTag(authTag)
  
  return Buffer.concat([
    decipher.update(ciphertext),
    decipher.final()
  ]).toString('utf8')
}
```

## What Gets Encrypted

Conar encrypts the following sensitive data:

### Database Credentials

- **Connection strings** - Full connection URI including passwords
- **Usernames** - Database user accounts
- **Passwords** - Database passwords
- **Host information** - Server addresses and ports

### Stored Data

- **Saved queries** - SQL queries saved for reuse
- **API keys** - Keys for AI providers (if stored locally)
- **Session tokens** - Authentication tokens

<Warning>
Plaintext credentials are NEVER stored on disk or in the database. Only encrypted versions are persisted.
</Warning>

## What is NOT Encrypted

### Non-Sensitive Data

These items are stored in plaintext for functionality:

- **Connection names** - Your friendly names for connections
- **Labels** - Connection labels (Production, Staging, etc.)
- **Colors** - UI color preferences
- **Database types** - PostgreSQL, MySQL, etc.
- **Query history metadata** - Timestamps and row counts (not query text)

### In-Memory Data

Data in application memory is not encrypted:

- **Active connections** - Currently open database connections
- **Query results** - Data returned from queries
- **Editor content** - SQL being written in the editor

<Note>
In-memory data is cleared when you close Conar or disconnect from a database.
</Note>

## Database Connection Security

### SSL/TLS Encryption

Conar supports encrypted database connections using SSL/TLS:

<CodeGroup>

```text PostgreSQL SSL
postgresql://user:pass@host:5432/db?sslmode=require
```

```text MySQL SSL
mysql://user:pass@host:3306/db?ssl={"rejectUnauthorized":true}
```

```text SQL Server Encryption
mssql://user:pass@host:1433/db?encrypt=true
```

</CodeGroup>

### SSL Certificate Support

For databases requiring custom SSL certificates:

```text
postgresql://user:pass@host:5432/db?sslmode=verify-full&sslcert=/path/to/client.crt&sslkey=/path/to/client.key&sslrootcert=/path/to/ca.crt
```

Conar reads and validates:

- **Client certificates** (`sslcert`) - Prove client identity
- **Client keys** (`sslkey`) - Private key for client cert
- **CA certificates** (`sslrootcert`) - Verify server identity

### SSL Modes

| Mode | Description | Security Level |
|------|-------------|----------------|
| `disable` | No SSL | ⚠️ Low - Unencrypted |
| `require` | SSL required, no verification | 🟡 Medium - Encrypted but not authenticated |
| `verify-ca` | Verify CA certificate | 🟢 High - Encrypted and server verified |
| `verify-full` | Verify CA and hostname | ✅ Maximum - Full verification |

<Tip>
For production databases, always use `verify-full` to ensure both encryption and server authentication.
</Tip>

### HTTPS Only

All communication with Conar's servers uses HTTPS:

- **TLS 1.2+** - Modern encryption protocols only
- **Certificate pinning** - Prevents man-in-the-middle attacks
- **Secure headers** - HSTS, CSP, and other security headers

### Firewall Considerations

Conar needs to connect to:

1. **Your database servers** - Direct connection from your machine
2. **Conar API** (if using cloud features) - For sync and AI

</Tip>

## Security Best Practices

### For Database Administrators

1. **Create dedicated users** - Don't use admin accounts with Conar
2. **Limit permissions** - Grant only necessary privileges
3. **Enable SSL/TLS** - Require encrypted connections
4. **Use strong passwords** - Minimum 16 characters, mixed case, numbers, symbols
5. **Rotate credentials** - Change passwords regularly
6. **Monitor access logs** - Review who's connecting and when
7. **IP whitelist** - Restrict connections to known IPs

### For Developers

1. **Never commit credentials** - Don't put connection strings in code
2. **Use environment-specific connections** - Separate dev/staging/prod
3. **Label production connections** - Make them visually distinct
4. **Enable read-only when possible** - Reduce risk of accidental changes
5. **Review dangerous queries** - Double-check DELETE/DROP/TRUNCATE
6. **Keep Conar updated** - Install security patches promptly

### For Teams

1. **Share connection patterns, not credentials** - Document connection setup
2. **Use team sync carefully** - Understand who can access shared connections
3. **Audit connection access** - Know who has access to what
4. **Implement least privilege** - Each team member gets minimal needed access
5. **Offboard process** - Remove access when team members leave
